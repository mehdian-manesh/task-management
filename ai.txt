البته باید در نظر داشت که ساختن ‌working day همان check-in است.
هر working day یک تاریخ و زمان check-in و check-out دارد متعلق است به یک کاربر. همچنین هر working day تعدادی report دارد. هر ریپورت دارای یک متن (حداکثر ۱۰۰۰ کاراکتری) است، یک نتیجه دارد که می‌تواند یکی از مقادیر زیر را داشته باشد:

هنوز تمام نشده
با موفقیت انجام شد
به تعویق افتاد
موفق به انجام آن نشدم
کنسل شد همچنین هر ریپورت متعلق است به یک task و هر task هم میتواند متعلق باشد به یک پروژه. کاربران عادی فقط دسترسی دیدن به پروژه‌ها را دارند و مدیریت آنها توسط ادمین انجام می‌شود. همچنین کاربران عادی فقط پروژه‌هایی را می‌بینند که ادمین مشخص کرده است. درمورد taskها کاربران عادی می‌توانند یک task را در حالت draft ایجاد کنند (گفتیم taskها می‌توانند متعلق به یک پروژه باشند یا نباشند. البته مسلما نمی‌توانند از پروژه‌ای که به آن دسترسی ندارند برای خود تسک تعریف کنند) و از این تسک در ریپورت خود استفاده کنند. این تسک زمانی که به تایید ادمین برسد تبدیل می‌شود به یک تسک تایید شده و دیگر draft نیست. ویژگی‌های دیگر تسک‌ها و پروژه‌ها عبارتند از نام، توضیح، رنگ، لیست افرادی که به آن تسک یا پروژه assign شده‌اند (می‌تواند این لیست خالی باشد، به صورت پیش‌فرض، لیست کاربرانی که به یک تسک assign شده‌اند برابر است با لیست کاربرانی که به کل پروژه assign شده)، تاریخ شروع، deadline، برآورد میزان نفر-ساعت مورد نیاز. همچینین تسک‌ها یک فیلد integer می‌توانند داشته باشند که مشخص می‌کند برای کدام فاز پروژه هستند. از آنجایی که بعدا باید یک کانبان برای برنامه در فرانت طراحی کنیم، تسک‌ها و پروژه‌ها می‌توانند یکی از وضعیت‌های زیر را داشته باشند: postpone, backlog, todo, doing, test, done, archive حالا بیا قدم به قدم بریم جلو و این برنامه رو بنویسیم. اگر پیشنهادی هم در خصوص این blueprint داری یگو
ریپورت‌ها حتما باید متعلق به یک تسک باشند (اگر تسک مورد نظر وجود نداشت یک draft به صورت خودکار براش ساخته بشه). فیلد‌های هر ریپورت میشه: پوزری که صاحبشه، لینک (کلید خارجی) به تسک، نتیجه (نتیجه می‌تونه به صورت خودکار روی نتیجه تسک هم اثر بذاره)، یک کامنت (که کاربر توضیح میده چیکار کرده برای این تسک، وارد کردنش اختیاریه)، تاریخ و زمان شروع به کار، تاریخ و زمان اتمام کار.
در مورد فیدبک‌ها هم باید بگم که کاربران عادی فقط دسترسی CRUD به فیدبک‌هایی که خودشون ساخته‌اند رو دارند. کاربر ادمین ولی همه رو می‌بینه. هر فیدبک شامل یک متن توضیحات اجباریه، بقیه فیلد‌ها اختیاری‌اند و عبارتند از: نوع (میتونه انتقاد، پیشنهاد و سوال باشه)

## System Overview
This is a full-stack task management application designed to track working hours and task reports. The system uses Django REST Framework for the backend and React for the frontend.

## Core Components

### Backend (Django + DRF)
- Custom User model
- JWT Authentication
- Nested routing with drf-nested-routers
- PostgreSQL database
- Timezone: Asia/Tehran

### Frontend (React)
- Material-UI with RTL support
- Redux Toolkit for state management
- JWT authentication
- Persian calendar support
- Responsive design

## Models

### User
- Standard Django user with is_staff flag for admin privileges

### WorkingDay
- user: ForeignKey to User
- check_in: DateTimeField (auto_now_add)
- check_out: DateTimeField (nullable)
- is_on_leave: BooleanField

### Report
- working_day: ForeignKey to WorkingDay
- task: ForeignKey to Task
- result: CharField (choices: ongoing, success, postponed, failed, cancelled)
- comment: TextField (max_length=1000)
- start_time: DateTimeField
- end_time: DateTimeField

### Task
- name: CharField
- description: TextField
- color: CharField
- project: ForeignKey to Project (optional)
- assigned_users: ManyToManyField to User
- start_date: DateField
- deadline: DateField
- estimated_hours: IntegerField
- phase: IntegerField
- status: CharField (choices: postpone, backlog, todo, doing, test, done, archive)
- is_draft: BooleanField

### Project
- name: CharField
- description: TextField
- color: CharField
- assigned_users: ManyToManyField to User
- start_date: DateField
- deadline: DateField
- estimated_hours: IntegerField
- status: CharField (same choices as Task)

### Feedback
- user: ForeignKey to User
- type: CharField (choices: criticism, suggestion, question)
- description: TextField
- created_at: DateTimeField

## API Endpoints

### Authentication
- POST /login/ - Login with username/password
- POST /logout/ - Logout user
- POST /token/refresh/ - Refresh JWT token

### Working Days
- GET /working-days/ - List working days
- POST /working-days/ - Check in
- POST /working-days/{id}/check_out/ - Check out
- POST /working-days/{id}/leave/ - Mark as leave
- GET /working-days/{id}/reports/ - List reports for working day
- POST /working-days/{id}/reports/ - Create report

### Tasks
- GET /tasks/ - List tasks
- POST /tasks/ - Create task
- GET /tasks/{id}/ - Get task details
- PATCH /tasks/{id}/ - Update task
- DELETE /tasks/{id}/ - Delete task

### Projects
- GET /projects/ - List projects
- POST /projects/ - Create project (admin only)
- GET /projects/{id}/ - Get project details
- PATCH /projects/{id}/ - Update project (admin only)
- DELETE /projects/{id}/ - Delete project (admin only)

### Feedbacks
- GET /feedbacks/ - List feedbacks
- POST /feedbacks/ - Create feedback
- PATCH /feedbacks/{id}/ - Update feedback
- DELETE /feedbacks/{id}/ - Delete feedback

## Key Features

### Task Workflow
1. Regular users can create draft tasks
2. Admin approves tasks, moving them from draft
3. Tasks can be associated with projects
4. Task status updates based on report results

### Working Day Management
1. Check-in creates a working day
2. Multiple reports can be added during the day
3. Check-out closes the working day
4. Leave can be marked without check-out

### Authorization Rules
1. Regular users:
   - View assigned projects
   - Create draft tasks
   - Manage own working days and reports
   - Manage own feedback

2. Admins:
   - Full project CRUD
   - Task approval
   - View all feedback
   - User assignment

## Conventions
1. All dates use Persian calendar (Shamsi)
2. Times are in Tehran timezone
3. All text content is in Persian
4. RTL layout throughout the application
